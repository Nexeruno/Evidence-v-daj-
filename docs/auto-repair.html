🔧 Safe Auto-Repair System

Inteligentní debugování a archivace bez smazání dat

🎯 Účel

Automaticky detekuje a opravuje chyby v datech bez jejich smazání.

  • Detekuje invalid transakce
  • Opravuje chybná metadata
  • Archivuje pro analýzu
  • Logguje vše pro audit

⏰ Frekvence

Běží automaticky každých 6 hodin.

  • 0:00 (půlnoc)
  • 6:00 (ráno)
  • 12:00 (poledne)
  • 18:00 (večer)

📊 Co se detekuje a opravuje?

1️⃣ Invalid PENDING Transakce

Detekuje pending transakce s chybějícími nebo neplatnými poli:

  • Bez názvu (missing_title)
  • Bez nebo se zápornou částkou (invalid_amount)
  • Bez kategorie (missing_category)
  • Bez typu nebo špatný typ (invalid_type)
  • Bez data (missing_date)
Akce: Archivuje do users/{uid}/archivedProblems s metadata. Zachovává všechna původní data!

2️⃣ Invalid RECURRING Transakce

Detekuje opakující se transakce s chybnou konfigurací:

  • Bez názvu
  • Se zápornou/nulovou částkou
  • Bez kategorie
  • S chybným typem (vydaj/prijem)
  • S chybnou frekvencí
Akce: Opravuje na validní výchozí hodnoty. Přidává repairHistory s detaily. NIKDY je neminusuje!

3️⃣ Staré PENDING Transakce

Detekuje pending transakce starší než 48 hodin:

  • Nevyžádané/zapomenuté pending
  • Zašuměné UI
Akce: Archivuje do archivedProblems pro pozdější analýzu.

📦 Archivované Kolekce

users/{uid}/archivedProblems

Obsahuje všechny archivované problematické transakce s metadaty:

{
  // Původní data transakce
  title: "Nákup",
  amount: 100,
  category: "Jídlo",
  type: "vydaj",

  // Metadata archivace
  archivedAt: Timestamp,
  reason: "INVALID_PENDING_TRANSACTION" | "OLD_PENDING_TRANSACTION_48H",
  validationErrors: ["missing_title", "invalid_amount"],
  originalId: "doc-id",
  userId: "uid"
}

systemRepairs (globální log)

Centrální log všech oprav provedených systemem:

  duration: 1234, // ms
  repairs: [
    {
      type: "ARCHIVE_INVALID_PENDING",

🛠️ Technické Detaily

Bezpečnostní Guardrails

NIKDY se neděje:
  • ❌ Smazání plateb (jen archivace)
  • ❌ Detekce duplikátů (příliš nebezpečné)
  • ❌ Smazání vydajů/příjmů
  • ❌ Tichá deaktivace aktivních recurring
  • ❌ Mazání bez zálohování

Úpravy Recurring Transakcí

Když se opravují opakující se transakce:

// Automaticky se opravují chyby
- Chybí category? → "Ostatní"
- Chybí type? → "vydaj"
- Chybná frequency? → 1
- Chybí title? → "Bez názvu"

// Vždy se zachovává:
- isActive stav
- lastGeneratedDate
- Celková struktura

// Přidá se:
- lastRepaired: timestamp
- repairHistory: [
    {
      timestamp,
      fixes: ["fixed_title", "fixed_category"]
    }
  ]

Výkon

  • Procesuje všechny uživatele paralelně (batching)
  • Jeden záznam ~ 5-10ms
  • Celková doba ~ 30-60 sekund pro 1000+ transakcí
  • Neblokuje aplikaci (background job)
